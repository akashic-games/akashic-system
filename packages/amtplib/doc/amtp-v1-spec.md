## 1. プロトコルの概要

Akashic Message Transfer Protocol(AMTP) は TCP や WebSocket 等の到達保証・順番保証される通信プロトコル上で、2 段に多重化された仮想通信路を構築し、高速にバイナリメッセージを送受信するためのアプリケーション層のプロトコルである。

### 1.1 用語

- コネクション
  - TCP や WebSocket 等の通信プロトコルの通信路。
- エンドポイント
  - コネクションにおけるクライアントまたはサーバ。
- ピア
  - データを受信又は送信する対象となるエンドポイント。
- Channel
  - コネクション上に確立される仮想通信路。
- プライマリ Channel
  - フレームにおいて、Channel ID の省略が可能な特殊な Channel。
- Pipe
  - Channel 上に確立される仮想通信路。
- Push Pipe
  - データを断続的に送信するための Pipe。
- プライマリ Push Pipe
  - データフレームにおいて、Pipe ID の省略が可能な特殊な Push Pipe。
- Request Pipe
  - リクエスト／レスポンス形式のデータを送信するための Pipe。
- プライマリ Request Pipe
  - データフレームにおいて、Pipe ID の省略が可能な特殊な Request Pipe。
- ペイロード
  - AMTP の制御情報を含まない送受信すべきデータの実体。
- フレーム
  - コネクション上を流れる可変長データ。AMTP の最小通信単位。
- データフレーム
  - パイプ上を流れるペイロードを含むフレーム。
- 制御フレーム
  - AMTP の各種制御を行うためのフレーム。
- 制御データ
  - 制御フレームに含まれる制御用のデータ。
- ラベル
  - Channel 又は Pipe 生成時に指定する文字列。

## 2. ハンドシェイク

ハンドシェイクはクライアントから開始され、サーバの応答により完了する。

以下は、ハンドシェイクからデータ送信までのシーケンスを示す。

```
      Client                                   Server
         |          Establish Connection          |
         |<-------------------------------------->|
         |                                        |
         |              AMTP Handshake            |
         |--------------------------------------->|
         |                 Accept                 |
         |<---------------------------------------|
         |                                        |
         |              Create Channel            |
         |--------------------------------------->|
         |                 Accept                 |
         |<---------------------------------------|
         |                                        |
         |        Create Push/Request Pipe        |
         |--------------------------------------->|
         |                 Accept                 |
         |<---------------------------------------|
         |                                        |
         |         Send Push/Request Data         |
         |--------------------------------------->|

```

## 2.1 クライアント

クライアントは、サーバに Open 制御フレームを送信する。

以下は、Open 制御フレームの制御データを示す。

```
+---------------------------------------------------------------+
|                   Protocol Identifier (32)                    |
+---------------+-----------------------------------------------+
|  Version (8)  |
+---------------+-----------------------------------------------+
|                        Random Bytes (32)                      |
+---------------------------------------------------------------+
```

- Protocol Identifier
  - Protocol Identifier は 16 進数表記で `0x414d5450` となる。
- Version
  - AMTP のバージョン（1〜255）を指定する。
- Random Bytes
  - ランダムなバイト列。

サーバからの Accept 制御フレームの受信により、ハンドシェイクが完了する。

## 2.1 サーバ

サーバは、クライアントからの Open 制御フレームへの応答として、Accept 制御フレームをクライアントに送信する。

以下は、Accept 制御フレームの制御データを示す。

```
+---------------------------------------------------------------+
|                    Protocol Identifier (32)                   |
+---------------------------------------------------------------+
|                     Random Echo Bytes (32)                    |
+---------------------------------------------------------------+
```

- Protocol Identifier
  - Protocol Identifier は 16 進数表記で `0x414d5450` となる。
- Random Echo Bytes
  - クライアントの送信した Open 制御フレームの Random Bytes。

## 3. Channel

Channel の生成は、クライアントのみ可能である。
クライアントは、サーバに Channel 制御フレームを送信する。

以下は、Channel 制御フレームの制御データを示す。

```
+---+-----------------------------------------------------------+
| P |                    Channel ID                             |
|(1)|                        (31)                               |
+---+-----------------------------------------------------------+
|                       Channel Label                           |
+---------------------------------------------------------------+
```

- P
  - プライマリ Channel にするか否かを示す 1 ビットのフラグ。プライマリ Channel とする場合は 1 となる。
- Channel ID
  - Channel を識別する 1 から始まる数値。
- Channel Label
  - Channel のラベル。最大 64 バイトの UTF-8 で符号化された文字列となる。本仕様でその解釈は定義されない。このフィールドが存在しない場合、Channel のラベルは空文字列として扱われる。

サーバは、Channel の生成を受け付けると Accept 制御フレームを応答としてクライアントに送信する。

### 3.1 プライマリ Channel

Channel が 1 つしか存在しない場合、その Channel はプライマリ Channel となる。
また、Channel 生成時にプライマリに指定された Channel はプライマリ Channel となる。
プライマリ Channel はコネクション上にひとつしか存在してはならない。

プライマリ Channel にデータフレームを流す場合、データフレームの Channel ID を指定してはならない。

エンドポイントは、Channel 制御フレームとその応答となる Accept 制御フレームにより、プライマリ Channel の同期をとる必要がある。

クライアントは、データフレームの送信において、Channel 制御フレーム送信後に、プライマリ Channel の切り替えを行わなければならない。

クライアントは、データフレームの受信において、Channel 制御フレームの応答となる Accept 制御フレーム受信後に、プライマリ Channel の切り替えを行わなければならない。

サーバは、データフレームの送信において、Channel 制御フレームの応答となる Accept 制御フレームをクライアントに送信後に、プライマリ Channel の切り替えを行わなければならない。

サーバは、データフレームの受信において、Channel 制御フレームの受信後に、プライマリ Channel の切り替えを行わなければならない。

### 3.2 Channel の破棄

Channel の破棄は、クライアントのみ可能である。
クライアントは、サーバに CloseChannel 制御フレームにより破棄要求を送信する。

以下は、CloseChannel 制御フレームの制御データを示す。

```
+---+-----------------------------------------------------------+
| X |                    Channel ID                             |
|(1)|                        (31)                               |
+---+-----------------------------------------------------------+
```

- X
  - 予約済みのフィールド。送信時は必ず 0x0 となり、受信時は無視される。
- Channel ID
  - 破棄する Channel の Channel ID。

サーバは、CloseChannel 制御フレーム応答として、Accept 制御フレームをクライアントに送信する。

クライアントは、CloseChannel 制御フレーム送信後、破棄要求を行った Channel 向けのデータフレームは無視しなくてはならない。また、破棄要求を行った Channel にデータフレームを流してはならない。
サーバは、CloseChannel 制御フレーム受信後、破棄要求が行われた Channel にデータフレームを流してはならない。

プライマリ Channel が破棄された場合は、Channel ID が最小となる Channel をプライマリ Channel とする。

エンドポイントは、Channel 生成時と同様に、CloseChannel 制御フレームとその応答となる Accept 制御フレームにより、プライマリ Channel の同期をとる必要がある。

## 4. Pipe

### 4.1 Push Pipe

Push Pipe は、サーバ又はクライアントからデータを断続的に送信するための Pipe である。
Push Pipe を生成したエンドポイントからのみデータの送信ができる。

#### 4.1.1 プライマリ Push Pipe

Channel 上にエンドポイントが生成した Push Pipe が 1 つしか存在しない場合、その Push Pipe はプライマリ Push Pipe となる。

クライアントが生成したプライマリ Push Pipe と、サーバが生成したプライマリ Push Pipe がそれぞれ 1 つずつ Channel 上に存在し得る。

また、Push Pipe 生成時にプライマリに指定された Push Pipe はプライマリ Push Pipe となる。

プライマリ Push Pipe にデータフレームを流す場合、データフレームの Pipe ID を指定してはならない。

プライマリ Push Pipe は Push Pipe の生成・破棄により、動的に変更される。

エンドポイントは、Channel 生成時と同様に、Pipe 制御フレームとその応答となる Accept 制御フレームにより、プライマリ Push Pipe の同期をとる必要がある。

### 4.2 Request Pipe

Request Pipe は、サーバ又はクライアントからリクエスト／レスポンスを送信するための Pipe である。
Request Pipe を生成したエンドポイントからのみリクエストの送信ができる。

リクエストを表すデータフレームには、リクエストを識別するための Request ID を必ず指定しなくてはならない。
レスポンスを表すデータフレームには、対応するリクエストの Request ID を必ず指定しなくてはならない。

Request ID は 1 から始まる数値で、クライアントが生成したリクエストの場合は奇数、サーバが生成したリクエストの場合は偶数となる。

#### 4.2.1 プライマリ Request Pipe

Channel 上にエンドポイントが生成した Request Pipe が 1 つしか存在しない場合、その Request Pipe はプライマリ Request Pipe となる。

クライアントが生成したプライマリ Request Pipe と、サーバが生成したプライマリ Request Pipe がそれぞれ 1 つずつ Channel 上に存在し得る。

また、Request Pipe 生成時にプライマリに指定された Request Pipe はプライマリ Request Pipe となる。

プライマリ Request Pipe にデータフレームを流す場合、データフレームの Pipe ID を指定してはならない。

プライマリ Request Pipe は Request Pipe の生成により、動的に変更される。

エンドポイントは、Channel 生成時と同様に、Pipe 制御フレームとその応答となる Accept 制御フレームにより、プライマリ Request Pipe の同期をとる必要がある。

### 4.3 Pipe の生成

Pipe の生成は、クライアント又はサーバで行うことが可能である。
Pipe 制御フレームを送信することで Pipe が生成される。

```
+---+-----------------------------------------------------------+
| P |                      Channel ID                           |
|(1)|                        (31)                               |
+---+-----------------------------------------------------------+
| R |                       Pipe ID                             |
|(1)|                         (31)                              |
+---+-----------------------------------------------------------+
|                          Pipe Label                           |
+---------------------------------------------------------------+

```

- P
  - プライマリ Pipe にするか否かを示す 1 ビットのフラグ。プライマリ Pipe とする場合は 1 となる。
- Channel ID
  - Pipe を生成する Channel を指す Channel ID。
- R
  - 生成する Pipe の種別を示す 1 ビットのフラグ。Push Pipe の場合は 0x0、Request Pipe の場合は 0x1 となる。
- Pipe ID
  - Pipe を識別するための 1 から始まる数値。
- Pipe Label
  - Pipe のラベル。最大 64 バイトの UTF-8 で符号化された文字列となる。本仕様でその解釈は定義されない。このフィールドが存在しない場合、Pipe のラベルは空文字列として扱われる。

エンドポイントは、Pipe の生成を受け付けると Accept 制御フレームを応答として送信する。

エンドポイントは、Accept 制御フレームを送信後、指定された Pipe 向けのデータフレームの受信に備えなければならない。

Channel 上に存在する Pipe は、Pipe を生成したエンドポイント、Pipe の種別、Pipe ID の組み合わせによってのみ一意に決定される。

### 4.4 Pipe の破棄

Pipe の破棄は、Pipe を生成したエンドポイントからのみ可能である。
エンドポイントは ClosePipe 制御フレームを送信する。

以下は、ClosePipe 制御フレームの制御データを示す。

```
+---+-----------------------------------------------------------+
| X |                    Channel ID                             |
|(1)|                        (31)                               |
+---+-----------------------------------------------------------+
| R |                      PipeID                               |
|(1)|                        (31)                               |
+---+-----------------------------------------------------------+
```

- X
  - 予約済みのフィールド。送信時は必ず 0x0 となり、受信時は無視される。
- Channel ID
  - 破棄する Pipe が存在する Channel の Channel ID を指定する。
- R
  - 破棄する Pipe の種別を示す 1 ビットのフラグ。Push Pipe の場合は 0x0、Request Pipe の場合は 0x1 となる。
- Pipe ID
  - 破棄する Pipe の Pipe ID を指定する。

エンドポイントは、ClosePipe 制御フレーム送信後、破棄した Pipe 向けのデータフレームを無視しなくてはならない。また、破棄した Pipe にデータフレームを流してはならない。

プライマリ Pipe が破棄された場合は、Pipe ID が最小となる Pipe をプライマリ Pipe とする。

エンドポイントは、Pipe 生成時と同様に、ClosePipe 制御フレームとその応答となる Accept 制御フレームにより、プライマリ Push/Request Pipe の同期をとる必要がある。

## 5. フレームフォーマット

### 5.1 データフレーム

```
+---+---+---+---+---+
| F | C | P | R | X |
|(1)|(1)|(1)|(1)|(4)|
+---+---+---+---+---+-------------------------------------------+
| X |                        Channel ID                         |
|(1)|                           (31)                            |
+---+-----------------------------------------------------------+
| X |                         Pipe ID                           |
|(1)|                           (31)                            |
+---+-----------------------------------------------------------+
|                         Request ID (32)                       |
+---------------------------------------------------------------+
|                           Payload Data                        |
+---------------------------------------------------------------+
```

- F
  - フレームの種別を表す 1 ビットのフラグ。データフレームの場合は 0 となる。
- C
  - Channel を指定するか否かを表す 1 ビットのフラグ。プライマリ Channel の場合は 0、プライマリ Channel 以外の場合は 1 となる。
- P
  - Pipe を指定するか否かを表す 1 ビットのフラグ。プライマリ Pipe の場合は 0、プライマリ Pipe 以外の場合は 1 となる。
- R
  - 応答が必要なデータフレーム、又は応答となるデータフレームか否かを表す 1 ビットのフラグ。Push Pipe のデータフレームの場合は 0、Request Pipe のデータフレームの場合は 1 となる。
- X
  - 予約済みのフィールド。送信時は必ず 0x0 となり、受信時は無視される。
- Channel ID
  - C が 1 の場合に存在するフィールド。データフレームが紐づく Channel ID を指定する。
- Pipe ID
  - P が 1 の場合に存在するフィールド。データフレームが紐づく Pipe ID を指定する。
- Request ID
  - R が 1 の場合に存在するフィールド。Request ID を指定する。
- Payload Data
  - ペイロード。サイズはペイロードにより異なる。

### 5.2 制御フレーム

```
+---+-----------+
| F |    Type   |
|(1)|     (7)   |
+---------------------------------------------------------------+
|                           ID (32)                             |
+---+-----------+-----------------------------------------------+
|                         Control Data                          |
+---------------------------------------------------------------+
```

- F
  - フレームの種別を表す 1 ビットのフラグ。制御フレームの場合は 1 となる。
- Type
  - 制御フレームの種別。
- ID
  - 制御フレームを識別する 1 から始まる数値。クライアントが生成した制御フレームの場合は奇数、サーバが生成した制御フレームの場合は偶数となる。
  - 応答を表す Accept/Deny 制御フレームの場合は、対応する制御フレームの ID を指定する。
  - 応答を必要としない場合は 0x0 を指定しなければならない。
- Control Data
  - 制御データを表す。サイズは制御フレームの種別により異なる。

## 6. 制御フレームタイプ

- Accept (type = 0x1)
  - ピアに送信される制御フレームへの受け入れ承諾応答。
- Deny (type = 0x2)
  - ピアに送信される制御フレームへの拒否応答。
- Open (type = 0x3)
  - クライアントからサーバに送信される AMTP 開始要求。
- Channel (type = 0x4)
  - クライアントからサーバに送信される Channel の生成要求。
- Pipe (type = 0x5)
  - ピアに送信される Pipe の生成要求。
- Close (type 0x6)
  - ピアに送信される AMTP 終了通知。
- CloseChannel (type = 0x7)
  - クライアントからサーバに送信される Channel の破棄要求。
- ClosePipe (type = 0x8)
  - ピアに送信される Pipe の破棄要求。
- Error (type = 0x9)
  - プロトコルエラー。

## 7. プロトコルエラー

何かしらの異常により、AMTP の接続を維持できない場合、Error 制御フレームをエンドポイントに送信する。Error 制御フレームの送信を以って AMTP の接続は終了となる。

この際、エラーメッセージとして、UTF-8 で符号化された文字列を制御データに格納することが出来る。本仕様でその解釈は定義されない。

Error 制御フレームを受信したエンドポイントは、AMTP の接続を即座に終了しなければならない。
